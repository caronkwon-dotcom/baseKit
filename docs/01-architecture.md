# 01. Architecture

## 1. 목표

BaseKit은 업무시스템 구축 시 반복적으로 필요한 관리자 화면과 공통 구조를 빠르게 조합할 수 있도록 하는 Admin Shell 기반 프론트엔드 레퍼런스 프로젝트이다.

단순한 예제 화면이 아니라, 실제 업무시스템에서 반복적으로 등장하는 다음 구조를 공통 패턴으로 정리하는 것을 목표로 한다.

* 공통 레이아웃
* 메뉴 구조
* 프로그램 진입 방식
* 탭 기반 작업 흐름
* 조회 / 목록 / 상세 / 액션 패턴
* 도메인 중심 화면 설계
* 향후 권한 및 백엔드 연계 확장 기반

---

## 2. 1차 범위

1차 범위는 **React + TypeScript + Mock Data** 기반으로 화면 골격과 공통 패턴을 확정하는 단계이다.

이 단계에서는 다음을 우선 반영한다.

* Header / Sidebar / Workspace 기반 공통 프레임
* MDI Tabs 기반 프로그램 작업 구조
* 사용자관리 화면 중심의 기준 패턴 수립
* 공통코드관리, 메뉴관리, 회사관리, 권한관리, 시스템설정 등 유사 화면 확장 기반 마련
* Mock Data 기반 목록 / 상세 / 검색 흐름 검증

1차 단계에서는 다음 항목은 구현 범위에서 제외하거나 최소화한다.

* Spring REST API 연동
* 실제 DB 연동
* 인증/인가 구현
* 저장/수정/삭제의 실제 서버 처리
* 상세 권한 정책 집행
* AI 매뉴얼 기능의 실제 생성 처리

---

## 3. 2차 확장 방향

2차 단계에서는 1차에서 확정한 화면 구조와 공통 패턴을 기반으로 실제 업무시스템 기능으로 확장한다.

주요 확장 방향은 다음과 같다.

* Spring REST API 연동
* 실제 데이터 저장/조회 처리
* 권한 체계 반영
* 메뉴/프로그램/액션 단위 접근 제어
* 공통 컴포넌트 고도화
* 화면별 실업무 기능 반영
* AI 기반 메뉴 매뉴얼 기능 확장

즉, 1차는 “화면 구조와 UX 패턴 검증”, 2차는 “실제 서비스 구조와 정책 반영”을 목표로 한다.

---

## 4. 기본 프레임 구조

BaseKit의 기본 화면 구조는 다음과 같다.

* Header
* Sidebar
* MDI Tabs
* Workspace

이 구조를 통해 사용자는 좌측 메뉴에서 프로그램을 선택하고, 선택된 프로그램은 탭 단위로 열린다. 열린 프로그램은 중앙 Workspace 안에서 작업하며, 여러 프로그램을 동시에 전환할 수 있다.

기본 구조는 아래와 같은 흐름을 따른다.

```text
Header
 ├ Top Navigation / 공통 정보 영역
 ├ 현재 위치 / 공통 액션 영역
Sidebar
 ├ 메뉴 탐색
MDI Tabs
 ├ 열린 프로그램 목록
Workspace
 ├ 현재 활성 프로그램 화면
```

이 구조는 단순 페이지 이동형 관리자 화면보다, 실제 업무시스템에서 자주 사용하는 다중 작업 방식에 더 적합하다.

---

## 5. MDI 기반 작업 구조

BaseKit은 일반적인 단일 라우팅 중심 관리자 사이트가 아니라, **MDI Tabs 기반 업무시스템형 구조**를 기본으로 한다.

핵심 원칙은 다음과 같다.

* 메뉴 클릭 시 해당 프로그램 탭을 연다.
* 동일 프로그램은 중복 탭을 만들지 않는다.
* 열린 탭은 활성화, 전환, 닫기가 가능하다.
* Workspace는 현재 활성 탭의 화면을 표시한다.
* 사용자 작업 흐름은 “메뉴 선택 → 탭 오픈 → 화면 작업” 순서로 진행된다.

이 결정은 BaseKit을 단순한 샘플 화면이 아니라, 실업무 운영 화면으로 확장하기 위한 핵심 구조이다.

---

## 6. 화면 패턴 기준

1차 화면 패턴 기준은 **사용자관리 화면**을 중심으로 잡는다.

기본 패턴은 다음과 같다.

* 페이지 제목
* breadcrumb 또는 현재 위치 표시
* 검색 조건 영역
* 목록 Grid 영역
* 상세 정보 영역
* 액션 버튼 영역

이 패턴은 향후 아래 화면에도 재사용한다.

* 공통코드관리
* 메뉴관리
* 회사관리
* 권한관리
* 시스템설정
* 기타 관리성 프로그램

즉, 사용자관리 화면은 단일 기능 화면이 아니라, BaseKit 전체 관리화면의 템플릿 역할을 한다.

---

## 7. UI 방향

BaseKit의 UI는 일반 소비자용 서비스 화면이 아니라, **업무시스템형 compact UI**를 기본 방향으로 한다.

정책은 다음과 같다.

* 기본 폰트는 과도하게 크지 않게 유지한다.
* 화면 여백은 넉넉함보다 정보 밀도를 우선한다.
* Grid와 실제 작업 영역이 화면의 중심이 되도록 한다.
* 검색 조건 영역은 과도하게 높지 않게 압축한다.
* 업무화면 특성상 탭, 버튼, 입력창, 테이블의 높이를 compact 기준으로 조정한다.

현재 1차 기준선은 다음과 같다.

* `.page h1 = 20px`
* 기본 본문은 14px 중심
* 보조 텍스트는 12~13px
* 업무 데이터가 많은 화면에서도 효율적으로 사용할 수 있도록 전체 밀도를 조정한다.

---

## 8. 메뉴 구조에 대한 현재 판단

현재 1차 구현에서는 Sidebar 중심의 메뉴 구조를 우선 적용하였다.
다만 최종 메뉴 UI는 아직 확정하지 않았다.

현재까지의 판단은 다음과 같다.

* 메뉴는 최종적으로 3 Depth까지 확장될 가능성이 높다.
* 1 Depth 메뉴는 향후 Top 영역으로 올리는 방향을 검토한다.
* 좌측 영역은 2~3 Depth 메뉴를 중심으로 재구성할 가능성이 있다.
* 메뉴 트리가 길어질 경우, 단순 좌측 트리만으로는 UI 효율이 떨어질 수 있다.
* 메뉴 슬라이딩, Top + Left 혼합 구조 등은 후속 UI 검토 항목이다.

즉, 현재 메뉴 구조는 1차 검증용이며, 최종 메뉴 UX는 별도 의사결정 대상으로 관리한다.

---

## 9. 데이터 및 명명 방향

BaseKit은 화면을 단순 UI 컴포넌트 묶음으로 보지 않고, 도메인 중심 구조를 유지한다.

예를 들어 사용자관리에서는 다음과 같은 업무 필드명을 사용한다.

* `USER_ID`
* `LOGIN_ID`
* `USER_NAME`
* `ROLE_CODE`
* `USE_YN`

도메인 타입, Mock Data, API 계약 계층은 업무 용어 중심의 명명 기준을 유지하고, React 내부 로컬 상태나 UI 이벤트 처리에는 camelCase를 병행할 수 있다.

이 기준은 화면과 데이터의 의미를 명확히 유지하기 위한 것이다.

---

## 10. 공통 컴포넌트 방향

BaseKit은 반복되는 UI 구조를 공통 컴포넌트로 분리해 재사용 가능하도록 설계한다.

현재 또는 향후 공통화 대상은 다음과 같다.

* AppLayout
* Header
* Sidebar
* MdiTabs
* Workspace
* PageHeader
* SearchPanel
* DataTable
* SummaryCard
* Detail Panel 계열 컴포넌트

다만 초기 단계에서는 과도한 추상화보다, 실제 화면에서 재사용되는 수준까지 먼저 정리한 뒤 점진적으로 공통화한다.

---

## 11. 향후 확장 후보

BaseKit은 향후 아래와 같은 공통 업무 모듈 확장을 고려한다.

* Grid 교체형 Adapter 구조
* 파일 업로더
* 에디터
* 템플릿 / PDF 출력
* Scheduler 관리
* 내부 결재
* 메일 / 템플릿
* 외부 Web Service 연계
* 호출 이력 관리
* 게시판
* AI 기반 메뉴 매뉴얼 기능

이들 기능은 현재 모두 구현하지 않더라도, 구조적으로 수용 가능한 방향을 유지한다.

---

## 12. 정리

BaseKit의 1차 아키텍처는 다음과 같이 요약할 수 있다.

* React + Mock Data 기반으로 공통 업무 화면 구조를 먼저 확정한다.
* Header / Sidebar / MDI Tabs / Workspace 구조를 기본 프레임으로 사용한다.
* 사용자관리 화면을 기준 패턴으로 삼아 다른 관리화면으로 확장한다.
* UI는 compact 업무시스템형 밀도를 기본으로 한다.
* 메뉴 구조와 권한 구조는 향후 확장 가능하게 설계한다.
* 2차에서 Spring REST, 권한, 실데이터, 매뉴얼 기능 등으로 확장한다.

BaseKit은 단순 관리자 샘플이 아니라, 실제 업무시스템 구축을 위한 프론트엔드 표준 기반을 만드는 것을 목표로 한다.
