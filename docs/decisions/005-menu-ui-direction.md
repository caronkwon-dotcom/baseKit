# 005. Menu UI Direction

## 상태

검토중

## 배경

BaseKit 1차 구현에서는 Sidebar 중심 메뉴 구조를 우선 적용하여 프로그램 진입, MDI Tabs, Workspace 흐름을 검증하였다.

이 방식은 초기 구현과 구조 검증에는 효과적이었으나, 실제 업무시스템 관점에서 다음과 같은 한계 가능성이 확인되었다.

* 메뉴가 늘어날수록 좌측 트리가 길어진다.
* 3 Depth 이상으로 확장될 경우 가독성이 떨어질 수 있다.
* 상위 업무군 구분이 약해질 수 있다.
* 화면 좌측 공간이 메뉴에 과도하게 사용될 수 있다.
* 긴 메뉴 구조에서 탐색 효율이 저하될 수 있다.

BaseKit은 단순 소규모 관리자 화면이 아니라, 향후 기능과 프로그램이 늘어나는 업무시스템형 구조를 목표로 하므로, 메뉴 UI 방향을 별도 검토 대상으로 관리할 필요가 생겼다.

---

## 현재 구현 상태

현재 1차 구현에서는 다음 구조를 사용한다.

* Sidebar 중심 메뉴
* 메뉴 클릭 시 프로그램 탭 오픈
* MDI Tabs + Workspace 구조
* 사용자관리 등 프로그램 화면 진입 확인

즉, 현재 Sidebar 기반 메뉴는 **구조 검증용 1차 구현**으로 본다.

---

## 판단

현재 Sidebar 구조만으로도 기본 동작은 가능하지만, 최종 메뉴 UI는 아직 확정하지 않는다.

향후 메뉴 구조는 다음 방향을 우선 검토한다.

* **1 Depth 메뉴는 Top 영역 배치**
* **좌측 영역은 2~3 Depth 메뉴 중심 구성**
* 긴 트리 구조에 대응할 보조 UI 검토
* 메뉴 슬라이딩 및 접기/펼치기 정책 검토

즉, 현재 Sidebar-only 구조를 최종안으로 고정하지 않고, **Top + Left 혼합 구조**를 유력 후보로 둔다.

---

## 검토 방향

### 1. 1 Depth Top 배치

1 Depth 메뉴는 상단 Header 또는 Top Navigation 영역으로 배치하는 방식을 우선 검토한다.

예시:

* 시스템관리
* 운영관리
* 기준정보관리
* 통계관리

이렇게 하면 상위 업무군을 명확히 구분할 수 있다.

장점:

* 상위 업무군 인지가 쉬움
* 좌측 메뉴 길이 축소 가능
* 화면 좌우 공간 활용이 효율적임
* 상위 카테고리 전환이 직관적임

---

### 2. 좌측 2~3 Depth 중심 구성

좌측 Sidebar는 선택된 1 Depth의 하위 메뉴만 보여주는 구조를 우선 검토한다.

예시:

```text id="ezlqfh"
Top: 시스템관리 / 운영관리 / 통계관리

Left (시스템관리 선택 시):
- 사용자관리
- 메뉴관리
- 공통코드관리
- 권한관리
- 시스템설정
```

또는 3 Depth가 필요한 경우:

```text id="h2rk0p"
Top: 운영관리

Left:
- 배치관리
  - 스케줄관리
  - 실행이력
- 연계관리
  - 인터페이스관리
  - 호출이력
```

이 구조는 하위 메뉴 집중도를 높이고, 좌측 탐색 영역의 복잡도를 줄이는 데 유리하다.

---

### 3. 메뉴 슬라이딩 및 열림 상태

메뉴가 많아질 경우 다음 기능을 검토한다.

* 하위 메뉴 접기/펼치기
* 활성 메뉴 강조
* 열림 상태 유지 여부
* 메뉴 슬라이딩 처리
* 좌측 폭 축소/확장
* 긴 메뉴명 처리

현재 차수에서는 이 동작을 최종 확정하지 않고, 후속 UI 검토 항목으로 둔다.

---

### 4. 긴 트리 구조 대응

업무시스템은 시간이 지나면서 메뉴가 길어지는 경우가 많다.
따라서 메뉴 UI는 짧은 구조에서만 예쁘게 보이는 방식으로 결정하면 안 된다.

특히 다음 상황을 염두에 둔다.

* 메뉴 수가 계속 증가하는 경우
* 특정 업무군 하위에 메뉴가 과도하게 몰리는 경우
* 3 Depth 구조가 자주 나타나는 경우
* 메뉴명 자체가 긴 경우
* 메뉴 권한에 따라 보이는 구조가 달라지는 경우

따라서 단순 Sidebar 트리만으로 고정하지 않고, **확장 시에도 버틸 수 있는 구조**를 선택하는 것이 중요하다.

---

## 현재 확정 / 미확정 구분

### 현재 확정된 사항

* 현재 차수는 Sidebar 기반으로 구조 검증 수행
* 메뉴 클릭 → MDI Tabs 오픈 → Workspace 표시 흐름 유지
* 프로그램 단위 작업 구조는 유지
* 메뉴는 최종적으로 3 Depth 확장 가능성을 전제로 함

### 아직 미확정인 사항

* 1 Depth Top 배치 최종 확정
* Sidebar-only 유지 여부
* Top + Left 혼합 구조 최종 채택 여부
* 메뉴 슬라이딩 방식
* 열림 상태 유지 정책
* 메뉴 검색/즐겨찾기/최근메뉴 기능 추가 여부

즉, 현재는 메뉴 UI의 최종 형태를 고정하지 않고, 구조 검증 이후 후속 검토 대상으로 유지한다.

---

## 영향

이 결정으로 인해 현재 구현은 다음 원칙을 따른다.

* 현재 Sidebar 구조는 사용하되, 최종 고정안으로 간주하지 않는다.
* 메뉴 관련 컴포넌트는 향후 Top + Left 구조로 재편할 수 있도록 유연성을 남긴다.
* Menu / Program / Action의 데이터 구조는 유지한다.
* 프로그램 오픈 단위와 MDI Tabs 구조는 계속 유지한다.

즉, 메뉴 표현 UI는 바뀔 수 있지만, 프로그램 실행 구조 자체는 유지하는 방향이다.

---

## 후속 검토 항목

다음 항목은 메뉴 UI 재검토 시 함께 본다.

* Top Navigation 시안
* Sidebar 폭 정책
* 메뉴 접기/펼치기 UX
* 3 Depth 표현 방식
* 긴 메뉴명 표시 정책
* 메뉴 검색 기능
* 즐겨찾기 / 최근사용 메뉴 도입 여부
* 권한별 메뉴 노출 변화 시 UX 영향
* 모바일/소형 해상도에서의 메뉴 동작

---

## 결론

현재 BaseKit의 Sidebar 기반 메뉴는 1차 구조 검증용 구현으로 유지한다.

다만 최종 메뉴 UI는 다음 방향을 유력 후보로 검토한다.

* 1 Depth는 Top 영역 배치
* 좌측은 2~3 Depth 메뉴 중심 구성
* 긴 메뉴 트리에 대응할 보조 UI 추가 검토
* 메뉴 슬라이딩 및 접기/펼치기 정책 후속 결정

본 문서는 메뉴 UI 최종 확정 전까지, 현재 구현과 향후 방향을 함께 관리하는 중간 결정문서로 사용한다.
