# 003. Admin Shell and MDI

## 상태

승인

## 배경

BaseKit은 단순한 단일 페이지 이동형 관리자 화면이 아니라, 실제 업무시스템에서 사용하는 작업 흐름을 수용할 수 있는 공통 프레임이 필요했다.

기존 관리자 템플릿 방식만으로는 다음 요구를 충분히 반영하기 어려웠다.

* 여러 프로그램을 동시에 열고 전환하는 작업 방식
* 좌측 메뉴를 통한 프로그램 탐색
* 화면 공통 프레임의 재사용
* 검색 / 목록 / 상세 / 액션 패턴의 공통화
* 실제 운영형 관리화면으로 확장 가능한 구조

이에 따라 BaseKit 1차 프론트엔드 구조는 단순 라우팅 중심 화면이 아니라, **Admin Shell + MDI Tabs + Workspace** 구조를 기본 프레임으로 채택하기로 하였다.

---

## 결정

BaseKit 1차 화면 구조는 다음 공통 프레임을 기본으로 사용한다.

* Header
* Sidebar
* MDI Tabs
* Workspace

프로그램은 메뉴 클릭 시 탭 단위로 열린다.
동일 프로그램은 중복 탭으로 열지 않고, 이미 열려 있는 탭을 활성화한다.
현재 활성 프로그램은 Workspace 영역에 표시한다.

즉, BaseKit의 기본 작업 흐름은 다음과 같다.

```text id="eio8ro"
Menu Click
→ Program Open
→ MDI Tab Activate
→ Workspace Render
```

---

## 세부 정책

### 1. 메뉴 진입 정책

* 좌측 메뉴에서 프로그램을 선택한다.
* 메뉴는 프로그램 진입 구조로 사용한다.
* 프로그램 오픈 단위는 탭 기준으로 관리한다.

### 2. 탭 생성 정책

* 메뉴 클릭 시 해당 프로그램 탭을 연다.
* 동일 프로그램이 이미 열려 있으면 새 탭을 만들지 않는다.
* 기존 탭을 활성화한다.

### 3. 탭 활성화 정책

* 열린 탭은 사용자가 자유롭게 전환할 수 있다.
* 현재 선택된 탭의 프로그램 화면을 Workspace에 표시한다.

### 4. 탭 닫기 정책

* 사용자는 열린 탭을 닫을 수 있다.
* 활성 탭을 닫을 경우 인접 탭 또는 기본 화면으로 활성 상태를 이동한다.

### 5. 기본 화면 정책

* Home 또는 기본 프로그램은 초기 진입 화면 역할을 한다.
* Home 탭의 닫기 허용 여부는 추후 세부 정책으로 확정 가능하나, 초기 화면 개념은 유지한다.

---

## 구현 기준 화면

1차 구현에서는 **사용자관리 화면**을 기준 패턴으로 삼아 검증하였다.

기준 화면 패턴은 다음과 같다.

* 페이지 제목
* breadcrumb 또는 현재 위치 표시
* 검색 조건 영역
* 목록 Grid
* 상세 정보 영역
* 액션 버튼 영역

이 패턴은 향후 다음 프로그램으로 재사용한다.

* 공통코드관리
* 메뉴관리
* 회사관리
* 권한관리
* 시스템설정

---

## 구현 결과 요약

이번 차수에서 다음 항목을 실제 반영하였다.

* Header / Sidebar / Workspace 구조 반영
* MDI Tabs 반영
* 사용자관리 프로그램 탭 오픈 구조 반영
* 검색 패널 반영
* 목록 / 상세 / 액션 패턴 반영
* compact UI 방향에 맞춘 1차 밀도 조정 반영

즉, 이번 차수에서 BaseKit은 단순 관리자 페이지 샘플이 아니라, **업무시스템형 Shell 구조**로의 1차 전환을 완료하였다.

---

## 제외한 항목

이번 결정 범위에는 다음 항목을 포함하지 않는다.

* 최종 메뉴 UI 확정
* 1 Depth Top 메뉴 최종 반영
* 메뉴 슬라이딩 정책
* 3 Depth 최종 표현 방식
* 실제 권한 엔진 연결
* API 연동
* AI 메뉴 매뉴얼 기능 구현

이 항목들은 후속 결정문서 또는 후속 차수에서 별도로 다룬다.

---

## 영향

이 결정을 통해 BaseKit의 이후 화면 개발은 아래 기준을 따른다.

* 단일 라우팅 페이지 추가보다 프로그램 탭 구조를 우선 고려한다.
* 신규 관리화면은 사용자관리 기준 패턴을 재사용한다.
* 공통 컴포넌트는 Shell 구조와 Workspace 흐름을 전제로 설계한다.
* 메뉴 구조, 권한 구조, 매뉴얼 기능도 프로그램 단위로 연결 가능하도록 유지한다.

즉, 이 결정은 단순 UI 선택이 아니라, BaseKit 전체 프론트엔드 구조의 기준선 역할을 한다.

---

## 후속 검토 항목

다음 항목은 본 결정 이후 별도 검토가 필요하다.

* Home 탭 세부 정책
* 탭 상태 저장/복원 여부
* 메뉴 UI 방향 재검토
* 1 Depth Top + 2~3 Depth Left 구조 가능성
* 메뉴 트리 길이에 대한 UI 대응
* 권한과 탭 오픈 제어 연결
* 프로그램 단위 AI 메뉴 매뉴얼 확장 지점

---

## 결론

BaseKit 1차 프론트엔드 구조는 **Admin Shell + MDI Tabs + Workspace**를 기본 구조로 채택한다.

이 결정은 다음 이유로 유지한다.

* 업무시스템형 다중 작업 흐름에 적합함
* 프로그램 단위 관리에 유리함
* 공통 화면 패턴 재사용에 적합함
* 향후 권한, 메뉴, 문서화 기능 확장과 연결 가능함

본 결정은 향후 BaseKit 화면 구조와 프로그램 진입 방식의 기준으로 사용한다.
