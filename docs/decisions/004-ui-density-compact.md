# 004. UI Density Compact

## 상태

승인

## 배경

BaseKit 1차 화면 구현 결과, 기본 구조와 MDI 기반 작업 흐름은 확인되었으나 초기 화면은 일반 관리자 템플릿에 가까운 밀도를 가지고 있었다.

이 상태에서는 다음 문제가 발생할 가능성이 높았다.

* 제목과 보조 문구의 비중이 큼
* 입력창과 버튼 높이가 큼
* 탭과 패널의 여백이 큼
* 검색 영역이 화면을 과도하게 차지함
* Grid와 실사용 데이터 영역이 상대적으로 작아짐
* 복잡한 업무화면에서 스크롤이 빨리 발생함

BaseKit은 일반 소비자용 서비스 화면이 아니라, 조회/등록/수정/승인/관리 기능이 밀집되는 **업무시스템용 UI**를 목표로 한다.
따라서 초기 단계에서 UI 밀도 기준을 명확히 정할 필요가 있었다.

---

## 결정

BaseKit의 기본 UI 밀도는 **compact**를 기준으로 한다.

즉, BaseKit은 넉넉한 마케팅형/관리자 템플릿형 UI보다, 실제 업무 데이터와 화면 활용 효율을 우선하는 dense UI 방향을 기본값으로 채택한다.

이번 차수에서는 특히 사용자관리 화면을 기준으로 아래 원칙을 반영하였다.

* `.page h1 = 20px`
* 기본 본문은 14px 중심
* 보조 텍스트는 12~13px
* 입력창, 버튼, 탭, 테이블 행 높이는 compact 기준으로 축소
* 카드/패널 패딩과 섹션 간 간격을 축소
* 검색영역보다 Grid와 실작업 영역에 더 많은 공간을 배분

---

## 세부 기준

### 1. 페이지 제목

* `.page h1`은 `20px`를 기준으로 한다.
* 기존보다 과도하게 큰 제목 표현은 사용하지 않는다.

### 2. 섹션 제목

* 카드/패널/섹션 제목은 `16px` 수준을 기본으로 한다.
* 페이지 제목과 명확히 구분되되, 화면을 과도하게 차지하지 않도록 한다.

### 3. 본문 및 일반 텍스트

* 기본 본문, 버튼 텍스트, 탭 텍스트, 폼 라벨은 `14px` 중심으로 맞춘다.

### 4. 보조 텍스트

* breadcrumb, 설명 문구, helper text, placeholder 등은 `12~13px` 수준을 사용한다.

### 5. Grid

* Grid 헤더와 본문은 `13~14px` 수준으로 유지한다.
* 데이터 가독성과 정보 밀도 사이의 균형을 맞춘다.

---

## 높이 및 여백 방향

BaseKit의 compact 기준은 폰트 크기뿐 아니라 컴포넌트 높이와 여백에도 적용한다.

### 1. 입력창 / 셀렉트 / 버튼

* 높이를 compact 기준으로 줄인다.
* 목표는 시각적 답답함이 아니라, 업무화면에서 더 많은 정보가 한 화면에 들어오도록 하는 것이다.

### 2. 탭 높이

* 탭은 과도하게 높지 않게 유지한다.
* 탭 텍스트 가독성을 해치지 않는 범위에서 compact하게 조정한다.

### 3. 패널/카드 패딩

* 패딩은 넉넉함보다 정보 밀도를 우선한다.
* 검색 영역, 요약 카드, 상세 영역은 불필요한 상하 여백을 줄인다.

### 4. 섹션 간 간격

* 섹션 간 간격을 과도하게 두지 않는다.
* 페이지 안에서 제목, 검색, Grid, 상세영역이 자연스럽게 이어지도록 한다.

---

## 검색영역과 Grid의 우선순위

업무시스템에서 실제 사용 비중이 가장 높은 영역은 대개 다음이다.

* 검색 조건
* 목록 Grid
* 상세 정보
* 액션 버튼

이 중에서도 화면 공간은 **Grid와 실제 데이터 확인 영역에 우선 배분**하는 것을 원칙으로 한다.

따라서 compact UI 기준에서는 다음을 우선한다.

* 검색영역은 필요 이상으로 높지 않게 구성
* 설명 문구는 최소화
* Grid가 화면의 중심 영역을 차지하도록 배치
* 상세영역도 과도하게 넓지 않게 유지

즉, BaseKit은 “보여주기 좋은 화면”보다 “업무하기 좋은 화면”을 우선한다.

---

## 적용 범위

이 결정은 사용자관리 화면에 우선 반영하였으며, 향후 다음 화면에도 동일 기준을 적용한다.

* 공통코드관리
* 메뉴관리
* 회사관리
* 권한관리
* 시스템설정
* 향후 추가되는 관리성 프로그램 전반

즉, compact UI는 특정 화면의 임시 스타일이 아니라, BaseKit 전체의 기본 화면 밀도 기준이다.

---

## 구현 원칙

UI 밀도 조정은 개별 화면마다 임시로 하드코딩하지 않고, 가능한 한 공통 기준으로 관리한다.

권장 방식:

* 공통 CSS 변수 사용
* 공통 컴포넌트 스타일 기준 통일
* PageHeader, SearchPanel, DataTable, Tabs, Form Control 공통 반영

즉, 화면마다 따로 줄이는 방식이 아니라, **BaseKit 전역 스타일 정책으로 관리**하는 것이 바람직하다.

---

## 제외한 항목

이번 결정 범위에는 다음 항목을 포함하지 않는다.

* 다크모드
* 반응형 모바일 최적화
* 사용자별 UI 밀도 선택 옵션
* 테마 전환 기능
* 최종 디자인 시스템 문서화

현재 단계에서는 업무시스템 기본 밀도를 compact로 통일하는 것이 우선이다.

---

## 영향

이 결정을 통해 BaseKit 이후 화면 개발은 다음 기준을 따른다.

* 신규 화면은 기본적으로 compact 기준에서 시작한다.
* 제목, 버튼, 입력창, 탭, Grid 높이는 과도하게 크게 잡지 않는다.
* 화면 구성 시 Grid와 실업무 영역 확보를 우선한다.
* 화면을 화려하게 보이게 하기보다 정보 효율과 반복 작업 편의성을 우선한다.

이 기준은 향후 프로그램 수가 늘어나고 화면이 복잡해질수록 더 중요한 역할을 하게 된다.

---

## 후속 검토 항목

다음 항목은 후속으로 검토할 수 있다.

* compact / comfortable UI 밀도 모드 분리 여부
* 고정형/반응형 화면 폭 정책
* 그리드 행 높이 세부 기준
* 폼 라벨 정렬 방식
* 상세영역 접기/펼치기 정책
* 고해상도 모니터 대응 기준

다만 현재 단계에서는 BaseKit 기본 UI 밀도를 compact로 유지하는 것이 우선이다.

---

## 결론

BaseKit의 기본 UI 밀도는 **compact**로 결정한다.

핵심 기준은 다음과 같다.

* `.page h1 = 20px`
* 기본 본문 14px 중심
* 보조 텍스트 12~13px
* 입력창, 버튼, 탭, 테이블 행 높이 축소
* 패널 패딩 및 섹션 간 여백 축소
* 화면 공간은 Grid와 실제 작업 영역에 우선 배분

본 결정은 BaseKit을 일반 관리자 템플릿이 아니라, 실제 업무시스템형 화면으로 유지하기 위한 UI 기준으로 사용한다.
