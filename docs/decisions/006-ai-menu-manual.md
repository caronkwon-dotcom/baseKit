# 006. AI Menu Manual

## 상태

검토중

## 배경

업무시스템에서는 사용자 매뉴얼이 자주 필요하지만, 실제 운영에서는 다음과 같은 문제가 반복된다.

* 초기 매뉴얼 작성 비용이 큼
* 화면이 변경되어도 문서가 현행화되지 않음
* PPT/PDF 매뉴얼은 배포 후 거의 읽히지 않음
* 메뉴별/기능별로 필요한 설명을 즉시 찾기 어려움
* 운영자와 사용자 사이에 화면 사용법 이해 차이가 누적됨

특히 메뉴와 프로그램이 많은 시스템에서는 화면 설명, 필드 설명, 버튼 설명, 권한 차이, 변경 이력 등을 지속적으로 정리하는 것이 쉽지 않다.

이에 따라 BaseKit에서는 메뉴/프로그램 단위의 **AI 기반 매뉴얼 생성 및 현행화 기능**을 향후 확장 기능으로 검토한다.

---

## 목적

AI 메뉴 매뉴얼 기능의 목적은 다음과 같다.

* 메뉴별 사용자 매뉴얼 초안 자동 생성
* 화면 변경 시 매뉴얼 현행화 지원
* 필드, 버튼, 권한, 변경 이력 기반 설명 생성
* 운영자/기획자/개발자가 공통 기준 문서를 더 쉽게 유지
* 사용자에게 메뉴 단위로 필요한 설명을 빠르게 제공

즉, 단순 문서 생성 기능이 아니라, **업무시스템 화면 설명을 계속 살아 있게 유지하는 기능**을 목표로 한다.

---

## 기본 개념

AI 메뉴 매뉴얼 기능은 **Menu / Program 단위**로 생각한다.

예를 들어:

* 사용자관리 매뉴얼
* 공통코드관리 매뉴얼
* 메뉴관리 매뉴얼
* 권한관리 매뉴얼

메뉴 또는 프로그램을 기준으로, 해당 화면의 다음 요소를 기반으로 매뉴얼 초안을 생성할 수 있다.

* 화면 개요
* 사용 대상
* 권한 차이
* 조회 방법
* 등록/수정/삭제 방법
* 주요 필드 설명
* 버튼 설명
* 주의사항
* 변경 이력
* FAQ

## 검토 대상 범위

현재 차수에서는 실제 AI 생성 기능을 구현하지 않는다.
이번 문서에서 정의하는 범위는 **향후 기능 확장을 위한 개념과 확장 지점**이다.

즉, 현재는 다음을 확정하지 않는다.

* 실제 AI 모델 호출 방식
* 프롬프트 상세 구조
* 매뉴얼 저장 포맷
* PDF 생성 방식
* 승인 워크플로우
* 버전 관리 상세 규칙

현재 문서는 “이 기능을 BaseKit에서 다룰 가치가 있으며, 어디에 붙일 것인가”를 정리하는 수준이다.

---

## 기본 흐름

향후 AI 메뉴 매뉴얼 기능의 기본 흐름은 아래와 같이 구상한다.

```text id="m14d0v"
메뉴 또는 프로그램 선택
→ [매뉴얼] 버튼
→ 화면/필드/버튼/권한/변경이력 수집
→ AI가 매뉴얼 초안 생성
→ 담당자 검토/수정
→ 저장
→ 사용자 배포 또는 문서 출력
```

즉, AI가 최종문서를 자동 확정하는 것이 아니라, **초안을 만들고 사람이 검토하는 구조**를 기본으로 한다.

---

## 기본 원칙

### 1. AI는 초안 생성 도구이다

AI 결과를 그대로 공식 문서로 확정하지 않는다.
담당자의 검토와 수정 과정을 전제로 한다.

### 2. 메뉴 단위 또는 프로그램 단위로 생성한다

전체 시스템 매뉴얼을 한 번에 만드는 것이 아니라, **메뉴/프로그램 단위**로 생성하고 누적한다.

### 3. 현행화가 가능해야 한다

한 번 생성하고 끝나는 문서가 아니라, 화면 변경 시 재생성/보완할 수 있어야 한다.

### 4. 구조화된 형식을 지향한다

자유서술형 긴 문서보다, 반복 가능한 구조를 가진 표준 매뉴얼 형식을 우선한다.

### 5. 권한 차이를 설명할 수 있어야 한다

같은 화면이라도 권한에 따라 보이는 버튼이나 가능한 행위가 다를 수 있으므로, 매뉴얼도 이를 반영할 수 있어야 한다.

향후 메뉴 매뉴얼의 기본 구조 예시는 다음과 같다.

* 등록

## BaseKit에서의 확장 지점

AI 메뉴 매뉴얼 기능은 다음 위치와 구조를 기준으로 확장할 수 있다.

### 1. Page Header 액션 영역

각 프로그램 화면 우측 상단 또는 Page Header 영역에 `[매뉴얼]` 버튼을 둘 수 있다.

### 2. Menu / Program 메타데이터

메뉴/프로그램 메타데이터에 매뉴얼 관련 속성을 둘 수 있다.

예시:

* `manualEnabled`
* `manualTemplateType`
* `manualVersion`

### 3. Action 코드 확장

향후 액션 코드에 다음을 포함할 수 있다.

* `MANUAL_VIEW`
* `MANUAL_GENERATE`
* `MANUAL_EDIT`
* `MANUAL_PUBLISH`

### 4. 변경 이력 / 화면 설명 연계

화면 메타데이터, 필드 정의, 변경 이력과 연결되면 더 나은 매뉴얼 초안 생성이 가능하다.

## 현재 구현에서의 위치

현재 BaseKit 1차 구현에서는 다음까지만 고려한다.

* 메뉴/프로그램 단위 확장 가능성 확보
* Page Header 액션 영역에 향후 버튼 추가 가능 구조 유지
* 문서/결정문서에 기능 개념 명시
* Menu → Program → Action 구조와의 연결 가능성 유지

즉, 현재는 실제 UI 버튼이나 생성 기능을 넣지 않고, **구조적으로 나중에 붙일 수 있는 상태**만 확보한다.

---

## 권한과의 관계

AI 메뉴 매뉴얼 기능도 일반 업무 기능과 동일하게 권한 체계에 연결될 수 있다.

예시:

* 일반 사용자는 `MANUAL_VIEW`만 가능
* 운영자는 `MANUAL_EDIT` 가능
* 관리자만 `MANUAL_GENERATE`, `MANUAL_PUBLISH` 가능

즉, 매뉴얼 기능도 별도 예외 기능이 아니라, Program / Action 권한 체계 안에서 다룰 수 있어야 한다.

---

## 향후 기대 효과

이 기능이 확장되면 다음 효과를 기대할 수 있다.

* 화면별 매뉴얼 작성 부담 감소
* 문서 현행화 비용 감소
* 화면 이해도 향상
* 운영자 교육 효율 증가
* 사용자 문의 감소
* 메뉴/프로그램 메타데이터 활용도 증가
* BaseKit 차별화 요소 확보

즉, BaseKit이 단순 화면 템플릿을 넘어서, **업무시스템 문서화까지 돕는 구조**로 확장될 수 있다.

* Menu / Program 단위 기능으로 정의
* Page Header 액션 영역을 확장 지점으로 고려
* MANUAL 관련 Action 코드 확장 가능성 유지

### 아직 미확정인 사항

* 실제 AI 모델 및 호출 구조
* 프롬프트 상세 설계
* 저장 포맷
* 버전 정책
* 승인 프로세스
* PDF/문서 배포 방식
* 변경 이력 자동 수집 방식

즉, 현재는 개념과 구조만 확정하고, 구현 세부는 후속 단계에서 설계한다.

## 결론

BaseKit은 향후 **AI 기반 메뉴/프로그램 매뉴얼 생성 기능**을 확장 후보로 유지한다.

핵심 방향은 다음과 같다.

* 메뉴/프로그램 단위로 매뉴얼 생성
* AI는 초안 생성
* 담당자가 검토/수정
* 화면 변경 시 현행화 가능
* 권한 체계와 연결 가능
* Page Header 및 Action 구조에 자연스럽게 확장 가능

본 문서는 해당 기능을 즉시 구현하기 위한 문서가 아니라, BaseKit 구조 안에서 이 기능을 어떤 방향으로 수용할지 정의하는 결정문서로 사용한다.
