# 03. Role & Permission

## 1. 목적

BaseKit의 권한 구조는 단순히 메뉴를 보이게 하거나 숨기는 수준이 아니라, **업무시스템에서 사용자·조직·법인·기능·행위 단위까지 확장 가능한 기준 체계**를 만드는 것을 목표로 한다.

현재 1차 범위에서는 권한관리 화면과 구조적 확장 가능성을 우선 확보하고, 실제 정책 집행은 2차 단계에서 구체화한다.

즉, 이번 문서는 “현재 구현된 권한 UI”보다 **향후 어떤 기준으로 권한을 설계할 것인지**를 정리하는 문서에 가깝다.

---

## 2. 기본 원칙

BaseKit 권한 구조는 다음 원칙을 따른다.

* 권한은 메뉴, 프로그램, 액션 단위로 확장 가능해야 한다.
* 단순 사용자 단위 권한만이 아니라 조직/법인 기반 권한을 수용해야 한다.
* 내부 사용자와 외부 사용자의 권한 체계가 다를 수 있음을 전제로 한다.
* 화면 접근 권한과 데이터 처리 권한을 분리해서 볼 수 있어야 한다.
* 조회, 등록, 수정, 삭제, 승인 등 행위(Action) 단위 제어가 가능해야 한다.
* 작성자 본인 수정/삭제 같은 예외 정책을 수용할 수 있어야 한다.

---

## 3. 권한 구조의 기본 단위

권한 구조는 다음 단위를 기준으로 확장한다.

### 3.1 사용자(User)

개별 사용자는 로그인 및 실제 행위 주체이다.
예외 권한, 개별 차등 권한, 테스트 계정, 관리자 계정 등은 사용자 단위에서 관리할 수 있다.

### 3.2 조직(Organization)

업무시스템에서는 부서, 팀, 본부 등 조직 단위 권한이 자주 필요하다.
동일한 역할이라도 조직에 따라 접근 범위가 달라질 수 있다.

### 3.3 법인/회사(Company)

멀티 법인 또는 고객사 구조가 있는 시스템에서는 법인 단위로 메뉴/데이터/권한 범위를 구분할 수 있어야 한다.

### 3.4 역할(Role)

역할은 권한 부여의 기본 묶음 단위이다.

예시:

* 시스템관리자
* 일반관리자
* 부서운영자
* 조회전용사용자
* 외부협력업체사용자

### 3.5 프로그램(Program)

실제 권한이 적용되는 화면 단위이다.
예를 들어 사용자관리, 메뉴관리, 공통코드관리 등은 프로그램 단위 권한 대상이 된다.

### 3.6 액션(Action)

프로그램 안에서 수행하는 행위 단위이다.

예시:

* SEARCH
* CREATE
* UPDATE
* DELETE
* EXPORT
* APPROVE
* MANUAL_VIEW
* MANUAL_GENERATE

---

## 4. 내부 사용자와 외부 사용자

BaseKit은 권한 구조 설계 시 **내부 사용자와 외부 사용자**를 분리할 수 있어야 한다.

### 4.1 내부 사용자

내부 사용자는 일반적으로 조직/부서/직무 기반 권한 체계를 가진다.

예시:

* 구매운영자
* 부서장
* 본부 관리자
* 시스템운영자

이 경우 권한은 역할과 조직, 법인 범위와 함께 결합될 수 있다.

### 4.2 외부 사용자

외부 사용자는 협력업체, 고객사, 파트너사, 위탁업체 등일 수 있으며, 내부 사용자보다 더 제한된 범위를 가진다.

예시:

* 협력사 담당자
* 고객사 운영자
* 공급업체 사용자

외부 사용자는 보통 다음 제약을 가진다.

* 접근 가능한 메뉴가 제한됨
* 조회 가능한 데이터 범위가 제한됨
* 등록/수정 가능한 데이터 범위가 제한됨
* 특정 승인/관리 기능은 제외됨

---

## 5. 권한 부여 기준

BaseKit에서는 권한 부여를 다음 우선순위 구조로 확장할 수 있도록 설계한다.

```text id="p0w4md"
법인 권한
→ 조직 권한
→ 역할 권한
→ 사용자 예외 권한
```

예를 들어 다음과 같은 시나리오를 지원할 수 있어야 한다.

* 특정 법인에는 시스템관리 메뉴 전체 허용
* 특정 조직에는 조회만 허용
* 특정 역할에는 등록/수정 허용
* 특정 사용자에게만 예외적으로 삭제 허용

이 구조는 실제 업무 환경에서 자주 필요한 “기본 권한 + 예외 권한” 모델을 수용하기 위함이다.

---

## 6. 메뉴 / 프로그램 / 액션 권한

권한은 다음 세 층위로 나누어 볼 수 있다.

### 6.1 메뉴 권한

사용자가 메뉴를 볼 수 있는지 여부를 제어한다.

예:

* 시스템관리 메뉴 숨김
* 운영관리 메뉴만 표시
* 특정 상위 메뉴 전체 차단

### 6.2 프로그램 권한

사용자가 특정 프로그램 화면에 접근 가능한지 제어한다.

예:

* 사용자관리 접근 가능
* 메뉴관리 접근 불가
* 시스템설정 화면 접근 관리자만 허용

### 6.3 액션 권한

프로그램 안에서 어떤 행위를 할 수 있는지 제어한다.

예:

* 조회 가능, 등록 불가
* 수정 가능, 삭제 불가
* 승인 기능 관리자만 허용
* 엑셀 다운로드 제한

즉, “메뉴가 보인다”와 “화면에 들어갈 수 있다”와 “화면에서 어떤 버튼을 누를 수 있다”는 분리해서 봐야 한다.

---

## 7. 작성자 본인 정책

업무시스템에서는 단순 역할 권한 외에도 **작성자 본인 정책**이 필요한 경우가 많다.

예시:

* 본인이 등록한 건만 수정 가능
* 본인이 등록한 건만 삭제 가능
* 본인 부서 데이터만 수정 가능
* 본인 요청 건만 취소 가능

따라서 BaseKit은 다음과 같은 정책을 수용할 수 있어야 한다.

* 역할 기반 허용
* 조직 범위 기반 허용
* 작성자 본인 여부 기반 허용
* 상태값 기반 허용

이 부분은 2차 권한 정책 구현 시 업무별로 구체화한다.

---

## 8. 화면 권한과 데이터 권한의 구분

권한 설계 시 자주 놓치기 쉬운 부분은 **화면 권한과 데이터 권한이 다를 수 있다**는 점이다.

예를 들어:

* 사용자관리 화면에는 들어갈 수 있지만, 특정 조직 데이터만 조회 가능
* 메뉴관리 화면 접근은 가능하지만, 수정은 본사 관리자만 가능
* 공통코드관리 화면 접근은 가능하지만, 삭제는 금지

즉, 권한은 단순히 “페이지 접근 허용 여부”만으로 끝나지 않는다.

BaseKit은 최소한 다음 레벨의 구분을 염두에 둔다.

* 화면 접근 권한
* 액션 수행 권한
* 데이터 범위 권한

---

## 9. 현재 1차 구현 범위

1차 구현에서는 실제 권한 엔진을 만들지 않고, **권한관리 화면 및 구조 수용 가능성**을 우선 확인한다.

현재 단계의 의미는 다음과 같다.

* 권한관리 화면 뼈대 확인
* 역할 기반 관리화면 확장 가능성 확인
* 메뉴 / 프로그램 / 액션 개념 분리 기반 마련
* 향후 백엔드 권한 정책 연결 전제 확보

즉, 현재 권한관리 화면은 운영 가능한 완성형이라기보다, **BaseKit에서 권한 영역이 어디에 연결될지 보여주는 구조적 시범 화면**에 가깝다.

---

## 10. 향후 구현 방향

2차 단계에서는 다음과 같은 방향으로 실제 권한 처리를 검토한다.

### 10.1 메뉴 권한 연결

* 로그인 사용자 기준 메뉴 노출 제어
* 상위 메뉴 / 하위 메뉴 권한 처리

### 10.2 프로그램 접근 제어

* 프로그램 ID 기준 접근 가능 여부 판단
* 탭 오픈 시 권한 검증 가능성 확보

### 10.3 액션 버튼 제어

* 등록 / 수정 / 삭제 / 승인 / 다운로드 버튼 노출 및 비활성화 처리
* 단순 숨김과 읽기전용 정책 구분

### 10.4 데이터 범위 제어

* 법인/조직/사용자 기준 조회 범위 제한
* 본인 건만 처리 가능한 정책 반영

### 10.5 관리 화면 제공

* 역할관리
* 사용자별 예외권한 관리
* 메뉴권한 관리
* 프로그램권한 관리
* 액션권한 관리

---

## 11. 권한과 메뉴 구조의 관계

권한 구조는 메뉴 구조와 매우 밀접하게 연결된다.

* 메뉴는 탐색 구조
* 프로그램은 실제 접근 제어 대상
* 액션은 세부 행위 제어 대상

따라서 메뉴 구조가 확정될수록 권한 모델도 정교해질 수 있다.
특히 향후 1 Depth Top 메뉴, 2~3 Depth Left 메뉴 구조로 변경될 경우에도, 권한 기준은 Menu / Program / Action 분리 구조를 유지해야 한다.

---

## 12. 권한과 AI 메뉴 매뉴얼의 관계

향후 AI 메뉴 매뉴얼 기능이 추가될 경우, 매뉴얼 접근 자체도 권한 대상이 될 수 있다.

예시:

* MANUAL_VIEW
* MANUAL_GENERATE
* MANUAL_EDIT
* MANUAL_PUBLISH

즉, AI 매뉴얼 기능도 일반 업무 기능과 동일하게 액션 단위 권한에 포함할 수 있어야 한다.

---

## 13. 현재 확정 / 미확정 구분

### 현재 확정된 사항

* 내부 사용자 / 외부 사용자 구분 가능성 반영
* 역할, 조직, 법인, 사용자 단위 확장 고려
* 메뉴 / 프로그램 / 액션 단위 권한 구조 채택
* 작성자 본인 정책 수용 방향 유지
* 권한관리 화면을 구조적 확장 포인트로 확보

### 아직 미확정인 사항

* 실제 권한 테이블 구조
* 권한 우선순위 상세 알고리즘
* 예외 권한 충돌 처리 방식
* 메뉴 숨김 vs 비활성화 정책
* 버튼 숨김 vs 읽기전용 정책
* 데이터 범위 제어 구현 방식

즉, 현재는 권한 철학과 확장 방향은 정리되었고, 실제 엔진과 정책 세부는 후속 단계에서 구체화한다.

---

## 14. 정리

BaseKit의 권한 구조는 단순 메뉴 노출 제어가 아니라, **업무시스템의 확장 가능한 접근 제어 기준**을 정의하는 것이다.

현재까지의 방향은 다음과 같이 요약할 수 있다.

* 권한은 메뉴, 프로그램, 액션 단위로 본다.
* 내부/외부 사용자 구분을 수용한다.
* 법인 / 조직 / 역할 / 사용자 예외 권한 구조를 고려한다.
* 작성자 본인 정책과 데이터 범위 제어를 수용한다.
* 1차에서는 권한관리 구조를 마련하고, 2차에서 실제 정책을 구현한다.

이 문서는 이후 메뉴 문서, 도메인 문서, 사용자관리/권한관리 화면, 백엔드 권한 설계와 연결되는 기준 문서로 사용한다.
