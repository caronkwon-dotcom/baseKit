# 02. Menu Structure

## 1. 목적

BaseKit의 메뉴 구조는 단순한 화면 목록 구성이 아니라, **업무 프로그램을 탐색하고 실행하는 기준 체계**로 정의한다.

메뉴는 사용자가 기능을 찾는 진입점이며, 프로그램은 실제 작업 단위이고, 액션은 프로그램 안에서 수행하는 행위이다.
따라서 BaseKit의 메뉴 구조는 다음 기준으로 관리한다.

```text
Menu → Program → Action
```

이 구조를 기준으로 화면 구성, 탭 오픈, 권한 확장, 문서화, 매뉴얼 기능까지 연결 가능하도록 한다.

---

## 2. 기본 개념

### 2.1 Menu

Menu는 사용자가 화면 진입을 위해 탐색하는 구조이다.
업무 기능을 분류하고 계층화하는 역할을 한다.

예시:

* 시스템관리
* 기준정보관리
* 운영관리
* 통계관리

### 2.2 Program

Program은 실제로 열리는 작업 화면 단위이다.
메뉴 클릭 시 열리는 대상이며, MDI Tabs의 탭 단위가 된다.

예시:

* 사용자관리
* 공통코드관리
* 메뉴관리
* 회사관리
* 권한관리
* 시스템설정

### 2.3 Action

Action은 프로그램 안에서 수행하는 업무 동작이다.

예시:

* SEARCH
* CREATE
* UPDATE
* DELETE
* EXPORT
* APPROVE
* MANUAL_VIEW
* MANUAL_GENERATE

즉, 메뉴는 탐색 구조이고, 프로그램은 화면이며, 액션은 행위이다.

---

## 3. 메뉴 구조 원칙

BaseKit의 메뉴 구조는 다음 원칙을 따른다.

* 메뉴는 프로그램 진입을 위한 구조로 사용한다.
* 동일 프로그램은 메뉴 경로가 달라도 하나의 프로그램 단위로 관리한다.
* 메뉴와 프로그램은 분리된 개념으로 유지한다.
* 프로그램 오픈 단위는 MDI Tabs 기준으로 관리한다.
* 액션은 프로그램 내부 정책 및 권한 확장 기준으로 사용한다.

이 원칙을 지키면 메뉴 구조가 바뀌더라도 프로그램과 권한 체계를 상대적으로 안정적으로 유지할 수 있다.

---

## 4. 현재 1차 구현 구조

1차 구현에서는 Sidebar 중심 구조를 우선 적용하였다.
좌측 메뉴에서 프로그램을 선택하면 해당 프로그램이 Workspace에 열린다.

현재 구조는 다음과 같은 흐름이다.

```text
Sidebar Menu Click
→ Program Open
→ MDI Tab Activated
→ Workspace Render
```

현재 반영된 주요 프로그램 예시는 다음과 같다.

* Home
* 사용자관리
* 공통코드관리
* 메뉴관리
* 회사관리
* 권한관리
* 시스템설정

이 구조를 통해 프로그램 단위 화면 전환과 다중 작업 흐름을 확인하였다.

---

## 5. MDI 기반 메뉴 동작 정책

BaseKit은 메뉴 선택 후 단순 페이지 이동이 아니라, **MDI Tabs 기반 프로그램 작업 흐름**을 기본 정책으로 한다.

### 5.1 메뉴 클릭

* 메뉴 클릭 시 해당 프로그램 탭을 연다.
* 프로그램이 이미 열려 있으면 새 탭을 만들지 않고 기존 탭을 활성화한다.

### 5.2 탭 활성화

* 열린 프로그램은 탭으로 전환 가능하다.
* Workspace는 현재 활성 프로그램 화면을 표시한다.

### 5.3 탭 닫기

* 사용자는 열린 프로그램 탭을 닫을 수 있다.
* 탭을 닫으면 인접 탭 또는 기본 화면으로 활성 상태를 이동한다.

### 5.4 기본 화면

* Home 또는 기본 프로그램은 초기 진입 화면 역할을 한다.
* Home 탭의 닫기 허용 여부는 추후 정책으로 명확히 확정할 수 있다.

이 정책은 BaseKit을 일반 웹사이트형 메뉴 구조가 아니라, 업무용 멀티작업 UI로 유지하기 위한 핵심 기준이다.

---

## 6. 메뉴 계층 방향

현재는 Sidebar 중심으로 구현하였으나, 실제 업무시스템 확장성을 고려하면 메뉴는 최종적으로 **3 Depth**까지 확장될 가능성이 높다.

예시 구조:

```text
1 Depth: 시스템관리
  2 Depth: 사용자관리
  2 Depth: 메뉴관리
  2 Depth: 공통코드관리

1 Depth: 운영관리
  2 Depth: 배치관리
    3 Depth: 스케줄관리
    3 Depth: 실행이력조회
```

따라서 현재의 단순 좌측 계층만으로는 장기적으로 메뉴 길이와 탐색 효율에 한계가 생길 수 있다.

---

## 7. 향후 메뉴 UI 방향 검토

현재까지의 판단으로는 최종 메뉴 UI는 아래 방향을 우선 검토한다.

### 7.1 1 Depth 메뉴는 Top 배치 검토

1 Depth 메뉴는 화면 상단 영역으로 올리는 방식을 우선 검토한다.

예시:

* 상단 Top Navigation에 1 Depth 노출
* 좌측 영역은 선택된 1 Depth 하위의 2~3 Depth 중심 구성

이 방식은 다음 장점이 있다.

* 좌측 트리 길이 축소
* 주요 업무군 구분이 명확해짐
* 깊은 메뉴 구조를 더 안정적으로 수용 가능

### 7.2 좌측은 2~3 Depth 중심

좌측 영역은 선택된 상위 메뉴에 대한 하위 기능 탐색 영역으로 활용한다.

예시:

* Top: 시스템관리 / 운영관리 / 통계관리
* Left: 사용자관리 / 메뉴관리 / 공통코드관리 / 권한관리

### 7.3 메뉴 슬라이딩/접기

메뉴 트리가 길어질 경우를 대비해 다음 기능을 검토한다.

* 좌측 메뉴 접기/펼치기
* 하위 메뉴 슬라이딩
* Depth별 열림 상태 관리
* 스크롤 및 고정 영역 분리

현재 1차 구현에서는 이 부분까지는 확정하지 않고, 후속 UI 검토 대상으로 둔다.

---

## 8. 메뉴 트리 길이에 대한 고려

업무시스템에서는 메뉴가 시간이 갈수록 길어지는 경우가 많다.
따라서 초기에 메뉴 트리 길이에 대한 UI 전략을 고려해야 한다.

주요 고려 항목은 다음과 같다.

* 좌측 메뉴 전체 높이 관리
* 긴 메뉴명 처리
* Depth가 깊어질수록 가독성 저하 방지
* 자주 사용하는 메뉴 접근성 확보
* 현재 위치 표시 명확화
* 메뉴 검색 기능 필요 여부
* 즐겨찾기 / 최근 사용 메뉴 확장 가능성

현재 시점에서는 메뉴 UI를 단순 트리로 고정하지 않고, 향후 확장 가능한 구조로 유지하는 것이 중요하다.

---

## 9. 사용자관리 화면 기준 패턴

현재 메뉴 구조 검증은 사용자관리 화면을 기준으로 가장 먼저 확인하였다.

사용자관리 프로그램은 다음 패턴을 기준 화면으로 삼는다.

* 페이지 제목
* breadcrumb 또는 현재 위치
* 검색 조건 영역
* 목록 Grid
* 상세 정보 영역
* 등록 / 수정 / 삭제 액션

이 패턴은 향후 다른 프로그램에도 공통적으로 재사용한다.

예시:

* 공통코드관리
* 메뉴관리
* 회사관리
* 권한관리
* 시스템설정

즉, 메뉴 구조 문서는 단순 메뉴 계층만 다루는 문서가 아니라, **프로그램 진입 후 어떤 화면 패턴이 이어지는지**까지 연결해서 이해해야 한다.

---

## 10. 메뉴 메타데이터 관리 방향

BaseKit에서는 메뉴 구조를 단순 JSX 하드코딩이 아니라, 향후 메타데이터 기반으로 관리할 수 있도록 설계한다.

예시 관리 대상:

* 메뉴 ID
* 상위 메뉴 ID
* 메뉴명
* 정렬순서
* 연결 프로그램 ID
* 사용 여부
* 아이콘
* 권한 대상 여부

이 구조를 두면 향후 다음 확장이 쉬워진다.

* 메뉴 권한 제어
* 메뉴 트리 관리 화면
* 메뉴 현행화
* 다국어 확장
* 메뉴 기반 AI 매뉴얼 기능

---

## 11. 권한 및 매뉴얼 기능과의 연결 가능성

메뉴 구조는 향후 다음 기능과 연결될 수 있다.

### 11.1 권한

* 메뉴 접근 권한
* 프로그램 접근 권한
* 액션 단위 권한

### 11.2 AI 메뉴 매뉴얼

* 메뉴 또는 프로그램 단위 매뉴얼 보기
* 화면별 설명 생성
* 변경 이력 기반 문서화
* 사용자용 가이드 제공

따라서 메뉴 구조는 단순 UI가 아니라, 권한과 문서화의 기준 정보로도 활용할 수 있어야 한다.

---

## 12. 현재 확정 / 미확정 구분

### 현재 확정된 사항

* Menu → Program → Action 구조 사용
* 프로그램은 MDI Tabs 단위로 관리
* Sidebar 중심 1차 구현
* 동일 프로그램 중복 탭 방지 정책
* 사용자관리 기준 패턴 확정

### 아직 미확정인 사항

* 1 Depth Top 배치 최종 확정
* 메뉴 슬라이딩 방식
* 3 Depth의 최종 표현 방식
* Home 탭 닫기 정책
* 메뉴 검색 / 즐겨찾기 / 최근메뉴 확장 여부
* 긴 메뉴 트리에 대한 최종 UI 정책

즉, 현재 메뉴 구조는 “작동하는 1차 기준”이며, 최종 메뉴 UX는 후속 설계 대상으로 둔다.

---

## 13. 정리

BaseKit의 메뉴 구조는 단순한 트리 표현이 아니라, **업무 프로그램 실행 구조의 기준 체계**이다.

현재까지의 방향은 다음과 같이 요약할 수 있다.

* 메뉴는 탐색 구조
* 프로그램은 실제 탭 오픈 단위
* 액션은 프로그램 내 행위 단위
* MDI Tabs는 BaseKit의 기본 작업 방식
* 사용자관리 화면은 프로그램 패턴의 기준
* 향후 메뉴는 1 Depth Top + 2~3 Depth Left 구조를 우선 검토
* 긴 메뉴 트리에 대응할 UI 정책은 별도 검토 과제로 유지

이 문서는 이후 권한 문서, 도메인 문서, 메뉴관리 화면 설계, AI 매뉴얼 기능과 연결되는 기준 문서로 사용한다.
